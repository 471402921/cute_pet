# B. 意图可读性信号

按需读:Step 3 审 "意图可读性" 时打开。

## 要拦的腐烂

代码做对了,但 **WHY 没传达**。半年后回来看(或换人/Agent 接手),看不出"为什么写成这样",就会:
- 不敢改(怕踩坑) → 模块僵化
- 乱改(没意识到隐藏约束) → 引入回归
- 重写(看不懂干脆推倒) → 浪费

代码注释不是为了"解释代码做了什么"(代码自己能说),是为了**保留决策**:为什么不那样写、有什么不能改、跟谁有协议。

## 怎么看出来

### 信号 1:有"魔法"但没注释

代码里有让人会停一下问"为什么"的写法,但没注释:
- 看似多余的检查(`if (x != null && x.isNotEmpty)` 而 `x` 是非空类型)
- 不寻常的常量(magic number 没说明,如 `delay: 380ms`)
- 反直觉的顺序(把 await A 放在 await B 之前,但语义上看 B 更急)
- 例外处理"什么都不做"(`catch (_) {}`)

这些都需要 1 行注释说"为什么"。

### 信号 2:注释只描述代码做了什么

```dart
// bad — 描述代码做了什么(代码自己能看)
// Increment counter by 1
counter++;

// bad — 引用具体任务/PR(会过期)
// Added per JIRA-1234
final timeout = 5000;

// good — 解释非显然的 WHY
// 后端在 503 后会重置计数器,所以这里要本地补偿
counter++;
```

判断标准:把注释删掉,代码读起来还顺吗?**顺** = 注释多余。**不顺** = 注释是必要的。

### 信号 3:命名模糊,藏了语义

- `data` / `info` / `result` / `manager` / `helper` / `util` 类名(藏了具体职责)
- 单字母变量在多行函数里(除 `i` 在循环、`e` 在 catch)
- bool 字段名不带 is/has/should(`new` / `enabled` 而非 `isNew` / `hasEnabled`)
- enum 值用缩写(`PetSpecies.sb` 而非 `shibainu`)

不是必须改,但要标记——读起来要在脑子里"翻译一道",就是 Agent 上下文成本。

### 信号 4:架构决定没留迹

代码里做了一个会让其他人困惑的设计取舍,但没留下任何说明,也没在 doc/ 里。例如:
- pet_page 用 `StatefulWidget` 而不是 `GetView`(因为 GameWidget 不能每次 rebuild 重建)——如果代码里没 1 行说明,新加 page 的人会以为"GetView 是 default,这里为啥不用",可能反向重构出 bug
- 某个 controller 用了 `Get.put` 而不是 `Get.lazyPut`——一定有原因,但不写下来下个人会随意改

判断:有"反 default 选择"出现的地方,**应该有一行注释或文档锚点**。

### 信号 5:doc/ 与代码脱钩

最重要的可读性信号。doc/architecture.md 或 doc/conventions.md 里写了一条规则,但代码里悄悄破例了(没在文档里说明例外)。结果是:**未来读者/Agent** 信文档去做事,但实际代码不那样;或者信代码去做事,但被审核者引文档拦下。

排查方式:抽查 conventions.md 的几条具体标准,grep 看代码是否遵守:
- §1 错误处理:catch 是否抛 Failure 子类
- §4 i18n:Text 字面量是否走 l10n.xxx
- §10 freezed:数据模型是否带 @freezed + abstract

发现脱钩的:**两选一**——要么改代码合规、要么改文档承认这是约定例外。**不能放任脱钩**(双源真理是腐烂温床)。

## 报告时怎么描述

每条问题:
- **第一反应** —— 写"我作为 Agent 读到这会想什么"
- **隐含成本** —— 这个不修,新人/AI 在哪种场景会卡住
- **修复方向** —— 加一行注释 / 重命名 / 文档化 / 改代码合规
