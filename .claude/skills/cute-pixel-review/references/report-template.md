# 报告输出指引

按需读:Step 4 输出报告时打开。本文件给**报告应该有的形态**,不给固定模板字面填空。

## 报告的目的

让用户**做判断**,不是看清单。所以:

- 每条问题给得起证据(文件 + 行号 / 可 grep 字符串),让用户能立即跳过去看
- 描述"作为读者第一反应是什么"+"这不修将来谁会被绊",不只描述"代码长这样"
- 修复方向一句话,**不写代码**

## 严重度

- **严重(必须修)**:架构铁律破坏 / 规范脱钩 / 安全 / 数据丢失风险
- **中等(应该修)**:可读性 / 注释缺位 / 命名漂移 / 测试缺失
- **提示(可选)**:重构机会 / 改进建议(必须能映射到 architecture 或 conventions 某条原则)

## 形态

报告按 3 个原则维度分组(A 边界 / B 可读 / C 摩擦),每条问题包含:

- **位置**:文件路径 + 行号
- **问题**:违反什么(具体到哪条铁律 / 哪节 conventions)
- **第一反应 / 未来场景**:作为读者会怎么想 / 接下来谁会被卡
- **修复方向**:一句话

## 输出长度

控制在屏幕一页能看完。问题多时,**只列最重要的 5-10 条**,告诉用户"还有更多,先修这些再扫"。报告不是清单,是判断。

## 顶部加两件事

报告开头放:
1. 范围(审了哪些文件)
2. `make analyze` + `make fmt-check` 的结果。如有 issue,**告诉用户先修这些**,因为细节噪音会盖过架构问题。

## 不报的东西

- `make analyze` / `make fmt-check` 已经能拦的细节(避免 print、import 排序、`dynamic`、相对 import 等)
- 自动生成的 `*.freezed.dart` / `*.g.dart` / `lib/l10n/app_localizations*.dart`
- 个人审美(必须能映射到 architecture 或 conventions 某条原则)
- 推测性问题("可能在某种边界条件下出问题"——除非能给出具体场景)
